## Turn 3 — Assistant  `2026-04-27 20:19:23 PDT`  _(in:1 out:400 cw:2,189 cr:32,509 cost:$0.0240)_

The component uses **dynamic per-operation queue selection** — clients are not statically bound to queues.

### Queue Pool Setup

At controller attach time, `QueuePairPool::allocate()` (`qpair.rs:155`) creates queues at four fixed depths: **4, 16, 64, 256**. These go from shallow (low-latency) to deep (high-throughput).

### Selection Strategy

`QueuePairPool::select_index(batch_size)` (`qpair.rs:256`) walks the queues from shallowest to deepest and picks the **first queue with enough available capacity** for the operation:

- **Async read/write**: `batch_size = pending_ops.len() + 1` — the client's current in-flight count drives the selection. A client with 0 pending ops lands on the depth-4 queue; a heavily loaded client escalates to deeper queues.
- **Batch submit**: `batch_size = ops.len()` — the entire batch is pinned to a single queue via `qp_idx_override`, preventing fragmentation across queues.
- **Sync operations**: always `select_qpair(1)`, so they target the shallowest queue with at least 1 free slot.

If no queue has enough room, it falls back to the deepest queue (depth-256).

### Why This Design

It exploits the latency/throughput tradeoff of NVMe hardware queues: shallow queues have lower latency (less head-of-line blocking), deep queues sustain higher throughput. Lightly-loaded clients automatically get the low-latency path, while heavy clients or large batches escalate to queues built for it.

## Turn 9 — Assistant  `2026-04-27 20:21:22 PDT`  _(in:1 out:629 cw:1,604 cr:39,729 cost:$0.0274)_

Here's the full completion path:

### 1. Submission: SPDK gets a callback context

When the actor submits an async read/write to SPDK, it creates an `AsyncIoContext` (`actor.rs:48`) from a slab pool and passes it to SPDK via `Box::into_raw`. The context carries the `client_id`, operation `handle`, and a raw pointer back to the actor's `async_completions` vec.

### 2. SPDK fires the C callback

When the NVMe hardware completes an IO, `spdk_nvme_qpair_process_completions()` invokes `async_completion_cb` (`actor.rs:112`). This callback:
- Reconstructs the `AsyncIoContext` from the raw pointer
- Checks the NVMe status code for success/failure
- Pushes an `AsyncCompletionEntry` (client_id + handle + result) into the `async_completions` vec
- Returns the context to the slab pool

### 3. Actor poll loop dispatches to clients

On every `handle()` tick (`actor.rs:396-438`), the actor:

1. **Polls all qpairs** — calls `process_completions(0)` on each queue, which triggers the callbacks above and fills `async_completions`.
2. **Drains completions** — swaps `async_completions` into a scratch buffer and iterates it.
3. **Routes by client_id** — finds the matching `ClientState`, removes the `PendingOp` from `pending_ops` (if already removed by abort/timeout, the completion is silently discarded).
4. **Sends via callback channel** — pushes a `Completion::ReadDone` or `Completion::WriteDone` into that client's `callback_tx` SPSC channel.

The client receives completions by reading from the other end of that SPSC channel (`callback_rx`).

### Edge cases

- **Timeouts** (`actor.rs:444`): `check_timeouts()` removes expired ops from `pending_ops` and sends `Completion::Timeout` on the callback channel. If the SPDK completion arrives later, it's discarded (handle already removed).
- **Aborts** (`actor.rs:774`): sends `Completion::AbortAck` immediately.
- **Controller reset** (`actor.rs:463`): cancels all pending ops across all clients with `Completion::Error`, then drains SPDK completions without forwarding them.

So the flow is: **SPDK hardware -> C callback -> actor-local vec -> actor poll loop -> per-client SPSC callback channel -> client**.
