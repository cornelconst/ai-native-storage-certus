  Question 2 of up to 5 вҖ” Domain & Data Model

  FR-006 says size classes are created dynamically when a file of a new size

  NVMe). Also, unbounded size class creation could lead to memory pressure from

  maximum number of distinct size classes?

  I/O. Capping size classes prevents unbounded bitmap proliferation while still
  supporting the "small number of distinct size classes" expectation stated in
  the spec.

вҸә Design Summary

  The metadata manager is a flat-namespace extent allocator: 64-bit keys map to
  contiguous disk extents. Writers follow a reserve вҶ’ write data вҶ’ publish/abort
   lifecycle, with key conflict detection deferred to publish time. A two-level
  allocator backs this: a buddy allocator carves disk into power-of-two slabs,
  and per-size-class slab allocators hand out fixed-size slots via bitmaps (O(1)
   alloc/free). All state lives in memory; durability comes from periodic
  full-index checkpoints written to doubly-linked metadata chunk chains with
  per-page CRC32, committed atomically by updating a superblock at LBA 0.
  Recovery reads the superblock, walks the chunk chain, and rebuilds bitmaps and
   buddy state from the deserialized index.

  Likely Problems

  1. Write lock during checkpoint blocks all operations. The concurrency model
  is a single RwLock. Checkpoint needs at minimum a consistent snapshot of the
  index + slab list, which means taking the write lock. At 1M extents (~20 MB
  serialized), even just cloning the HashMap under write lock adds a latency
  spike visible to all concurrent reserve/publish/lookup callers. The research
  dismisses the actor model but doesn't address this specific contention window.
   You'll want a snapshot-then-release strategy, but that interacts awkwardly
  with problem #2.

  2. Metadata chunk allocation during checkpoint is a chicken-and-egg. Writing a
   new checkpoint requires allocating fresh metadata chunks from the metadata
  slab вҖ” but that allocation modifies the slab state, which is itself part of
  what's being checkpointed. If you snapshot-then-release (to fix problem #1),
  you need to pre-allocate chunks before snapshotting so the slab state is
  consistent at snapshot time. On checkpoint failure, those pre-allocated chunks
   must be explicitly freed in memory (on-disk they're invisible since the
  superblock wasn't updated). This sequencing isn't trivial and the plan doesn't
   call it out.

  3. Full-index write amplification scales poorly. Every checkpoint rewrites the
   entire index regardless of how much changed. At 1M extents and 5-second
  intervals, that's ~20 MB every 5 seconds вҖ” ~4 MB/s of sustained metadata I/O
  even during steady-state operation. The research correctly notes this is
  acceptable at the target scale, but it puts a ceiling on growth and wastes
  bandwidth on mostly-idle systems (the dirty-flag skip helps only when nothing
  changed at all).

  4. No slab reclamation or defragmentation. Once the buddy allocates a slab,
  it's permanent вҖ” even if every slot in it is freed. A workload that creates
  many files of size X, deletes them all, then creates files of size Y will
  accumulate dead slabs that can't be reused for the new size class. Over time
  with churn across different size classes, usable disk space shrinks even
  though the slabs are mostly empty.

  5. WriteHandle Drop safety. The RAII abort-on-drop requires the WriteHandle to
   hold a callback or reference back to the component's internal state (the slab
   bitmap). If the component's Arc is dropped while handles are outstanding вҖ”
  unlikely but possible in teardown races вҖ” the abort callback could access
  freed state. The implementation will need either Weak references (with silent
  no-op on upgrade failure) or ensure the component Arc is kept alive by
  outstanding handles (which prevents shutdown).

  6. Buddy allocator wastes tail space. Real disk sizes are rarely powers of
  two. The buddy allocator rounds down to the largest power-of-two that fits,
  leaving the remainder unusable. On a 1 TB disk with 4 KiB minimum bucket, you
  lose up to ~50% of the last buddy level's coverage (though in practice NVMe
  sizes are close enough to powers of two that this is usually small).

  ---
  Problems #1 and #2 are the ones most likely to cause real implementation
  headaches. I'd recommend deciding the checkpoint snapshot strategy before
  generating tasks вҖ” it cuts across the state, checkpoint, and slab modules.
