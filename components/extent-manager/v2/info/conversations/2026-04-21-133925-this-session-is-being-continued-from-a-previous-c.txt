     Superblock in Metadata Slab ‚ÄĒ Full-Disk Buddy Management

     Context

     Currently the first 4096 bytes of disk are reserved for the superblock,
     outside buddy-managed space. Regions start at byte 4096. This wastes a full
      buddy block (potentially very large ‚ÄĒ e.g., 1 GiB minimum rank) for just
     4096 bytes of superblock data. The fix: make the entire disk buddy-managed,
      pre-create one metadata slab per region at format time, and place the
     superblock in slot 0 of region 0's metadata slab.

     Changes

     1. Disk layout ‚ÄĒ regions start at byte 0

     src/lib.rs ‚ÄĒ format() and initialize()

     Remove SUPERBLOCK_SIZE from layout calculations:

     Old: usable_size = disk_size - SUPERBLOCK_SIZE
          base = SUPERBLOCK_SIZE + i * region_bytes
     New: usable_size = disk_size
          base = i * region_bytes

     Region 0 starts at byte 0. The superblock at LBA 0 is now inside region 0's
      buddy-managed space.

     2. Pre-create metadata slab per region at format time

     src/lib.rs ‚ÄĒ format()

     After creating each region's buddy, before wrapping in RegionState:

     let mut buddy = BuddyAllocator::new(base, size, params.block_size);
     let metadata_offset = buddy.alloc(params.slab_size as u64)
         .ok_or_else(error::out_of_space)?;

     let mut region = RegionState::new(i, buddy);
     let metadata_slab = Slab::new(metadata_offset, params.slab_size,
     params.chunk_size);
     let slab_idx = region.slabs.len();
     region.slabs.push(metadata_slab);
     region.size_classes.add_slab(params.chunk_size, slab_idx);

     if i == 0 {
         // Slot 0 = superblock
         region.slabs[slab_idx].mark_slot_allocated(0);
     }

     Region 0's first buddy alloc returns byte 0 (since the free list starts
     there), so the metadata slab starts at byte 0, and slot 0 is at byte 0 =
     LBA 0. The superblock write to LBA 0 remains unchanged.

     3. Recovery ‚ÄĒ mark superblock slot after reconstruction

     src/lib.rs ‚ÄĒ initialize()
